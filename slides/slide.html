You can't train modern robot foundation models on a laptop, so we use HPC. High Performance Computing (HPC) = a shared cluster of powerful nodes you reach over the network and use through a job scheduler.

Containers on HPC

A container is a box that holds a program and everything it needs to run — the right Python, the right CUDA libraries, system packages, and your own dependencies. Everything is inside one box. A container is not the same as a virtual machine. It shares the computer's kernel, so it starts very fast and runs at full speed.

The main idea: your setup lives inside the container, not on the node. So it works the same way everywhere — on your laptop, on the login node, and on any of the 1,120 compute nodes!

Why we must use containers on HPC:

  • You are not root. On a shared cluster, you can't run apt-get install to add system libraries. The container already has them inside.
  • Always the same. The same image gives you the same setup on every node, every time. Nothing changes from one run to the next.
  • No live installs at run time. The nodes can reach the internet, but you still don't pip install inside a job — that would make each run depend on the network and on whatever versions resolve that day. You build everything into the image once, so every node runs the exact same, pinned environment.

Docker vs. Apptainer

You've probably met Docker. Docker relies on a root-privileged background daemon (dockerd) — on a shared, multi-user supercomputer that is a security non-starter, so Docker isn't available here. HPC uses Apptainer (formerly Singularity) instead:

              DOCKER (laptops / CI)              APPTAINER (HPC)
──────────────────────────────────────────────────────────────────────────────
run model     containers via a root daemon       containers as your own process
              (dockerd)                          (no daemon)
image         layers in a system store           one file you own
              (/var/lib/docker)                  (image.sif)
host files    isolated; bind with -v             $HOME / cwd / /tmp auto-bound;
                                                 add more with -B
GPU           --gpus (NVIDIA Container Toolkit)  --nv (binds the host driver)
  • An Apptainer image is a single file, image.sif, sitting on shared storage. One file = your whole environment: copy it, version it, share it with the class.
  • You can still reuse the Docker ecosystem: apptainer pull docker://nvcr.io/... converts a Docker / NGC image into a .sif.
    †That's essentially what env/build_image.sh does for us.

Submitting jobs

A job is a request to the scheduler (PBS): you describe the resources you need and the commands to run, and PBS runs them for you on a compute node — not on the login node where you type your commands. You can run a job in one of two ways.

Interactive vs. batch

Interactive qsub -I ...Batch qsub job.pbs
What you getA shell on the node, so you can type commands liveIt runs on its own; you can close your laptop
Best forTesting, first setup, "can my container see the GPU?"Real training, long jobs, running many at once

A good habit: test interactively first, then put the commands that worked into a batch script.

Three commands = 90% of daily life

  • qsub: submit a job (prints a job id, e.g. 12345.opbs)
  • qstat: check status (Q = queued, R = running)
  • qdel: cancel a job (qdel 12345)

Batch script

A PBS script has two parts: directives (#PBS ..., read by the scheduler) and commands (a normal shell script, run after you get a node).

#!/bin/bash
#PBS -q <queue>             # which queue (this sets your GPU slice)
#PBS -W group_list=<group>  # which project to charge (yours is gw13)
#PBS -l select=1            # reserve 1 node / 1 GPU slice
#PBS -l walltime=00:15:00   # max run time; the job is killed when time is up
#PBS -N warmup              # job name (also used to name the output files)

cd "$PBS_O_WORKDIR"
module load <apptainer>
apptainer exec --nv image.sif python train.py     # the actual work

A note on walltime: be careful in both directions. Too short, and the job is killed before it finishes. Too long, and it waits longer for a free slot and uses more of your allocation. Make a good guess, then add a little extra.

Two values you fill in yourself

  • q <queue> — the GPU slice you are allowed to use. List the queues your group can submit to, then paste the name in.
  • group_list=gw13 — your project for this class.

Common #PBS directives

You write these as #PBS ... lines in the script (or as flags after qsub on the command line). The first five are the ones you almost always set.

DirectiveWhat it does
-q <queue>which queue to run in (sets your GPU slice)
-W group_list=gw13which project to charge
-l select=1how many nodes / slices to reserve
-l walltime=00:15:00maximum run time
-N <name>job name
-J 1-10run as a job array (here, 10 sub-jobs)
-o <file> / -e <file>rename the output / error file
-j oeput stdout and stderr in one file
-m abe + -M <address>email you when the job aborts / begins / ends

What happens after qsub

qsub job.pbs  ─►  [Q] queued  ─►  [R] running  ─►  [done]
                                                     │
                         warmup.o12345  ← stdout  ◄──┤
                         warmup.e12345  ← stderr  ◄──┘

The files are named <job name>.o<id> and .e<id>. Your print() output and any errors land there after the job finishes. You read them with ordinary shell commands — for a job named warmup with id 12345:

ls warmup.*               # warmup.o12345  warmup.e12345
cat warmup.o12345         # everything your program printed (stdout)
cat warmup.e12345         # error messages, if any (stderr)
tail -n 20 warmup.o12345  # just the last 20 lines

2. Imitation Learning with LeRobot

Imitation Learning System

Imitation learning: collect demonstrations of a task, then train a policy to copy them. The policy maps what the robot sees to what it should do.

A policy is a function: observation in, action out.

   observation                         action
 ┌───────────────┐                ┌─────────────────┐
 │ camera images │ ─► π(a | o) ─► │ joint / EE cmds │
 │ joint states  │    (policy)    │                 │
 └───────────────┘                └─────────────────┘

Why learn from demonstrations? Writing the controller by hand does not scale to messy tasks like grasping. Reinforcement learning can learn them, but it needs a reward function and a lot of trial and error — slow, and unsafe on a real robot. Imitation learning avoids both: show the task a few times, and the policy learns from the examples.

The basic recipe is behavior cloning. Treat it as supervised learning: each (observation, action) pair from the demos is one training example, and the policy learns to output the expert's action.

Every imitation learning system has the same four parts:

   ┌────────────────┐   teleop or scripted demos
   │ 1. DATA        │   (observation, action) pairs, many episodes
   └───────┬────────┘
           │
   ┌───────▼────────┐   the function π(a|o) to learn
   │ 2. POLICY      │   ACT, Diffusion Policy, VLA, ...
   └───────┬────────┘
           │
   ┌───────▼────────┐   fit the policy to the demos
   │ 3. TRAINING    │   loss, optimizer, GPU loop
   └───────┬────────┘
           │
   ┌───────▼────────┐   run the trained policy, measure success
   │ 4. EVALUATION  │   in a simulator or on a real robot
   └────────────────┘

1. Data — the demonstrations. Each episode is a trajectory of (observation, action) pairs from an expert. Observation = camera images + robot state. Action = the command at that step. Many episodes per task.

2. Policy — the function π(a|o). ACT, Diffusion Policy, VLA, and so on. Most predict a short chunk of future actions at once, not a single step. That is action chunking — what ACT is named after.

3. Training — supervised learning. Show the policy an observation, train it to output the expert's action, minimize the error. A normal loop: batches, loss, optimizer, GPU.

4. Evaluation — a low training loss does not mean the task is solved. Run the policy, measure its success rate. Test in a simulator (cheap, repeatable) or on a real robot.

At evaluation and deployment, the policy and the environment run in a loop:

   ┌──────────────── rollout loop ─────────────────┐
   │                                               │
   o ─►  policy π  ─►  a  ─►  environment  ─►  o' ─┘
                             (sim or real)

Behavior cloning is not as easy as it looks. Two problems shape every real system.

Compounding error — the policy only saw the expert's states. At test time it makes a small mistake, drifts into a state the demos never showed, and does not know what to do there.

Multimodality — there is often more than one correct action. You can go left or right around the cup. Train the policy to output one average action, and it averages left and right into a move that hits the cup.

A good policy also has to generalize. It should handle new object positions, and in multi-task setups follow a language instruction — not just replay the demos.

In practice it is a loop: collect data → train → evaluate → study the failures → collect more. You repeat until the success rate is good enough.

The five parts also have to fit together:

  • the action format in the data = what the policy outputs
  • the policy's output = what the environment accepts
  • the observations line up at every stage

Each lab used to build these parts itself. They rarely fit together, and results were hard to reproduce.

LeRobot Repository Overview

In the past, most robot learning labs built their own tools — their own data format, training loop, policy code, and evaluation code. Shared data formats already existed, but a shared platform did not. LeRobot solves this. It gives everyone one data format, a shared set of policies and evaluation environments, and one set of commands, so you can switch the policy or the dataset without rewriting your code.

 Without a standard                  With LeRobot
 ──────────────────                  ────────────────────────────────
  data format   ┐                     ┌─ LeRobotDataset  (one format)
  training loop ┤  incompatible       ├─ policies: ACT / DP / pi0 / ...
  policy code   ┤  silos per lab      ├─ CLIs: lerobot-train / -eval / -record
  eval harness  ┘                     └─ Hugging Face Hub (share data & weights)

LeRobot ships all five parts in one repository. Each part is a package, and they all share the same interfaces, so they fit together by default:

lerobot/
└─ src/lerobot/
   ├─ datasets/    [1 DATA]        LeRobotDataset — the one format 
   ├─ policies/    [2 POLICY]      ACT, Diffusion, pi0, SmolVLA, GR00T, ...
   ├─ optim/       [3 TRAINING]    optimizers and schedulers
   ├─ configs/     [3][4]          training and evaluation configuration
   ├─ scripts/     [3][4]          the CLIs: lerobot-train / -eval / -record
   ├─ envs/        [5 ENVIRONMENT] simulators: LIBERO, MetaWorld, ...
   ├─ robots/      [5 ENVIRONMENT] real hardware: SO-100/101, Koch, Reachy2, ...
   └─ processor/                   shared pre/post-processing (normalize, etc.)

So the whole pipeline lives in one package. The dataset class, the policy, the simulator, and the robot driver all speak the same interfaces. The ACT you train today on LIBERO data is the same policy class you would put on a real SO-101 arm — the policy and training code do not change, only the dataset and the environment do.

The three CLIs you meet today map onto the data → train → eval pipeline:

 lerobot-record                      lerobot-train             　   lerobot-eval
        |                                   |                              |
        v                                   v                              v
 collect data  ->  LeRobotDataset  ->  train policy  ->  checkpoint  ->  measure
  • lerobot-train — train a policy on a dataset (the main command today).
  • lerobot-eval — run a trained checkpoint and measure success rate.
  • lerobot-record — collect demonstrations from a robot (we only mention it today).

Everything is driven by command-line config (one flag per option), so an experiment is fully described by its command — easy to log, share, and reproduce.

Models Implemented in LeRobot

Every policy here does the same job — imitation learning. Take an observation, output an action chunk (the next H actions). What differs is how they make that chunk. Get that one idea and the rest is just three families.

 observation = images + state [+ language]
        │
        ▼
 action chunk = a_t, a_t+1, ...   (the next H actions)

 How the chunk is made:
   1. regress it directly (+ a latent)   ─► ACT
   2. denoise it from noise (iterative)  ─► Diffusion Policy
   3. pretrained VLM + action generator  ─► VLA family

The third family is the only one that takes a language instruction — that is what lets you tell it what to do. ACT and Diffusion Policy condition on images + state only, so each trained policy does one task.

Heavier as you go down the list:

 light
   │   ACT               regress chunk
   │   Diffusion Policy  denoise chunk
   │   SmolVLA           small VLA, single GPU
   │   π0 / π0.5         open-world generalist
   ▼   GR00T N1.5        cross-embodiment
 heavy

Each model is detailed below.

ACT — Action Chunking with Transformers

The lightweight default, from the ALOHA project. An encoder–decoder Transformer that predicts a whole chunk of actions at once instead of one step at a time. Predicting a chunk shortens the effective horizon, which is the main fix for compounding error in behavior cloning.

What the model is made of:

  • Image encoder — ResNet. Each camera view goes through a ResNet (ResNet-18) into tokens. Multiple views are supported (ALOHA uses four).
  • Joint state — a token. The follower's current joint positions enter as their own token.
  • Transformer encoder–decoder. The encoder fuses the image tokens, the state, and the latent z (below). The decoder takes a set of query embeddings — one per future timestep — and emits the whole action chunk in one pass. Output shape = [chunk_size, num_joints].

ACT is trained as a Conditional VAE, which is how it copes with the fact that human demos are not consistent — the same task is done slightly differently each time.

  • During training a small BERT-like encoder takes the demo's action chunk + joint state (with a [CLS] token) and squeezes it into a latent z — a "style variable" that captures which way this demo did it. The decoder then reconstructs the chunk from the observations + z.
  • Two losses. Action reconstruction (predicted vs. true chunk) + a KL term that keeps z close to a Gaussian prior.
  • At test time the encoder is thrown away and z = 0 (the mean). So it runs deterministically, but it has already learned the variation during training.

Temporal ensembling:

At inference ACT can predict a new chunk every timestep, so the chunks overlap. Temporal ensembling averages those overlapping predictions with an exponential weighting (recent predictions weigh more) → smooth motion, no jerky jumps between chunks. It is optional and adds little compute.

When to use it:

  • No language. It conditions on images + state, not instructions — single-task.
  • Fast to train, small. One forward pass per chunk, no denoising loop, no big VLM.
  • Best with one task and clean demos. That combination is exactly today's setup — which is why ACT is the policy we train first.

Diffusion Policy

This policy generates the action chunk by denoising: start from pure noise and refine it, step by step, into an action sequence — each step conditioned on the observation. It is behavior cloning, but the action head is a diffusion model (a DDPM).

How it works:

  • Training. Take a real action sequence, add Gaussian noise at a random step k, and train the network to predict the noise that was added. That is the whole objective — noise prediction.
  • Inference. Start from pure noise and run the reverse process: at each step the network predicts the noise, you subtract it, and the sample gets closer to a clean action sequence. Repeat for K steps (DDPM is many steps; faster samplers cut this down).
  • Conditioning. The observation is not denoised — it is a condition. Images go through a visual encoder once per step, and the features are injected into the denoiser; in the CNN version via FiLM (a learned scale-and-shift on the features). Keeping the observation out of the denoised data speeds up inference and suits real-time control.

The backbone — two choices:

  • 1D temporal U-Net (CNN). Robust, works out of the box on most tasks. Weakness: it prefers smooth, low-frequency signals, so it struggles when the action must change quickly and sharply.
  • Transformer (DDPM). Less over-smoothing, better for fast-changing actions, but needs more tuning.

Chunking and receding horizon:

It predicts a horizon of actions, executes only the first few, then re-plans — warm-starting the next prediction with the leftover actions from the last one. This is the same action-chunking idea as ACT, and it keeps the motion smooth across re-plans.

Strengths and costs:

  • Strength: it is naturally multimodal. Because it samples from noise rather than regressing one answer, it keeps the distinct modes (go left or right) instead of averaging them into a bad middle. This is where it shines on rich, varied behaviors.
  • Cost: it runs several denoising steps per chunk, so inference is heavier than ACT's single forward pass.
  • No language in the base form. It conditions on images and state, not instructions — single-task, not a generalist.

SmolVLA

LeRobot's small, open VLA. The idea is to make a VLA you can actually train and run yourself — about 450M parameters, trained only on community datasets, and built to run on a single (consumer) GPU. In practice this is the most realistic VLA for you to fine-tune.

Same two-part shape as π0, but smaller and tuned for speed:

  • A compact VLM backbone — SmolVLM-2. It builds on SmolVLM-2, which uses a SigLIP vision encoder to extract image features. It reads the images, the language instruction, and the robot state, and turns them into one set of tokens.
  • A flow-matching action expert. A small transformer that takes the backbone's features and predicts a chunk of actions — the same flow-matching idea as π0.

Where the speed comes from:

  • Layer skipping. The action expert reads features from only the first half of the VLM's layers, which roughly halves the backbone cost. Early layers already carry enough general meaning for control.
  • Fewer visual tokens. It feeds only the global image (no tiling) and uses a pixel-shuffle step to cap each frame at about 64 tokens.
  • Interleaved attention. The action expert alternates cross-attention (to look back at the VLM's features) and self-attention (over the action tokens), which keeps it both grounded and cheap.

When to use it:

Reach for SmolVLA when you want a real VLA you can fine-tune and deploy on modest hardware — not when you need the absolute strongest generalist (that is π0 / π0.5) or the fastest single-task policy (that is ACT). For this hands-on, it is the one you can actually train end-to-end yourself.

π0 / π0.5

These policies are built on top of a pretrained VLM (an internet-scale vision + language model). So the model already understands scenes and can follow a language instruction before it ever sees a robot.

π0 has two parts that share one attention stack:

  • A VLM backbone — PaliGemma. A pretrained model from Google: a SigLIP image encoder (~400M) plus a Gemma language model (2B), about 3B together. π0 does not train this from scratch — it starts from the released PaliGemma weights (the SigLIP ViT included). SigLIP turns each image into tokens; Gemma reads them together with the text and the robot state.
  • An action expert (~300M). A separate, smaller network, randomly initialized, that processes the state and action tokens. It takes the backbone's features and produces a continuous action chunk.

The two are fused into one network and interact only through attention (a mixture-of-experts style): the VLM half handles semantics and language, the action half handles control.

The dataflow:

 image  ─► SigLIP ─► image tokens ┐
 text   ─► text tokens            ├─► PaliGemma backbone
 state  ─► MLP ─► state token     ┘     │ features
                                        ▼
                  action expert ─► action chunk
                  (flow matching, ~10 steps)
  1. The image goes through SigLIP and becomes image tokens.
  1. The language instruction is tokenized.
  1. The robot state (joint angles, gripper) is projected into the same token space.
  1. The PaliGemma backbone reads all of them and produces features. This is
    where the web knowledge sits — it already knows what a "cup" and a "sink" are.
  1. The action expert takes those features and generates the next chunk of
    actions with flow matching.

Predicting a chunk, not one step, is the same action-chunking idea as ACT.

Flow matching vs. diffusion: Both turn random noise into an output over several steps. The difference is how.

  • Diffusion adds noise to the data in many small steps, then learns to undo it; at each step it predicts the noise to remove. Sampling follows a stochastic path (an SDE) and usually needs many steps.
  • Flow matching learns a velocity field: at each point between noise and data, it predicts which direction to move. Training is simple regression — take a point on the straight line x = τ·action + (1−τ)·noise and predict the direction action − noise. Sampling follows a deterministic path (an ODE) and needs fewer steps.

For a robot this matters. Flow matching needs about 10 steps instead of dozens, so actions come out fast and smooth — good for high-frequency control 👍

π0 vs. π0.5:

  • π0 — cross-embodiment. Trained on diverse robot data. Strong general manipulation.
  • π0.5 — tuned for open-world generalization. Co-trained on mixed data (many robots, web data, and high-level subtask labels next to low-level actions), so it transfers to scenes it never saw. It is roughly hierarchical: first decide what to do (a subtask), then how (the actions).

Both are heavy and language-conditioned. Reach for them when you need semantics and breadth — not when you want the fastest single-task policy (that is ACT).

†More recently, models like π0.7 have appeared, improving the steerability of VLAs — their language-following and in-context learning.

GR00T N1.5

NVIDIA's cross-embodiment foundation model (humanoids and arms). The design follows "thinking fast and slow" — a dual-system architecture inspired by human cognition.

 System 2 : a VLM          slow, semantic reasoning  (what to do)
        │  vision-language features
        ▼
 System 1 : action module  fast, high-Hz control     (how to move it)

Two systems that run at different speeds:

  • System 2 — a VLM (slow, semantic). Powered by NVIDIA's Eagle VLM, it reads the images and the language instruction and works out what to do. It is kept frozen, so its internet-scale knowledge stays intact during robot training.
  • System 1 — an action module (fast, motor control). A Diffusion Transformer (DiT) trained with action flow matching that figures out how to move. It alternates self-attention over the state and action tokens with cross-attention to the VLM's vision-language tokens, then an embodiment-specific decoder outputs the actions.

How it stays cross-embodiment:

Embodiment-specific encoders and decoders embed each robot's state and action space into one shared token dimension, so the same policy can be fine-tuned across different bodies — a single arm, two arms, or a humanoid.

How it is trained, and what N1.5 changed:

It is trained on a data pyramid — real captured data, synthetic data from the Isaac GR00T Blueprint, and internet-scale video. Over the original N1, N1.5 mainly improved language-following and generalization:

  • The VLM is kept frozen during both pretraining and finetuning (preserving language understanding), and was upgraded to Eagle 2.5 with better grounding and physical understanding.
  • A FLARE loss (aligning to future latent representations) was added alongside flow matching, which improved performance and unlocked learning from human videos, plus a modified MLP connector between the VLM and the DiT.
  • The payoff: strong language following (93.3% on GR-1 language tasks) and better zero-/few-shot adaptation to new objects and embodiments.

It is heavy — a generalist you adapt by post-training, not something you train from scratch.

†N1.5 → N1.7 is, roughly: a stronger reasoning VLM in System 2, and a lot more human-video pre-training feeding System 1's dexterity — on top of the larger action transformer that arrived in N1.6.

Which one — and the deeper split

 single task, clean demos, fast iteration     -> ACT
 multimodal behaviors, no language needed     -> Diffusion Policy
 need language / generalization, 1 GPU        -> SmolVLA
 max generalization + semantics, have compute -> π0 / π0.5, GR00T

The main difference behind all of this: specialist vs. generalist. ACT and Diffusion Policy are usually trained from scratch for a single task. The VLAs (π0, SmolVLA, GR00T) are almost never trained from scratch — you take the pretrained generalist and fine-tune it.

We use ACT today because it is the quickest way to get a finished job — but everything you do here (chunking, the training loop, evaluation) is the same as what you will use to fine-tune any of these models.

Structure of LeRobotDataset

A LeRobotDataset stores one recording as three coordinated layers. Splitting them keeps tables small and video compressed, while a single index ties them together.

 one LeRobotDataset
 ┌────────────────────────────────────────────────────────────┐
 │ Tabular  (parquet) │ states, actions, timestamps, indices  │
 │ Visual   (mp4)     │ camera frames, video-encoded          │
 │ Metadata (json)    │ features, fps, stats, tasks, etc.     │
 └────────────────────────────────────────────────────────────┘
            │  dataset[i] assembles frame i across all layers
            v
   { "observation.state": ..., "observation.images.front": ...,
     "action": ..., "timestamp": ..., "task": "..." }

Key fields you will actually use:

  • features — the schema: every key, its dtype and tensor shape.
  • fps — sampling rate; lets you convert "frames" to "seconds".
  • normalization stats — per-feature mean/std, used to normalize inputs during training.
  • delta_timestamps — pull neighboring frames (in seconds, relative to now) stacked on a new time axis T. This is how ACT/DP consume a short window of context.
 delta_timestamps = { "observation.state": [-2*dt, -dt, 0.0] }   (dt = 1/fps)

   frame:   i-2     i-1      i
            └──── window ────┘   ->  shape gains a leading T (=3)

We use the v3.0 dataset format (paired with lerobot 0.5.1).

Understand the LeRobot Format

A LeRobotDataset stores one dataset as three coordinated layers, tied together by metadata:

Tabular (parquet): states, actions, timestamps, episode/frame indices
Visual (mp4): camera frames, video-encoded (one stream per camera)
Metadata (json+parquet): features, fps, normalization stats, task strings

On disk (v3.0), it looks like this — note that many episodes are packed into a few chunked files, not one file per episode:

 <root>/
 ├── meta/
 │   ├── info.json                 # schema: features, shapes, dtypes, fps, codebase version
 │   ├── stats.json                # per-feature mean/std/min/max
 │   ├── tasks.parquet             # task strings  <->  integer task ids
 │   └── episodes/
 │       └── chunk-000/
 │           └── file-000.parquet  # per-episode info: length, task, offsets
 ├── data/
 │   └── chunk-000/
 │       └── file-000.parquet      # frame-by-frame state/action/index
 └── videos/
     └── observation.images.front/ # one folder per camera key
         └── chunk-000/
             └── file-000.mp4      # the camera frames, video-encoded

The schema lives in features. Which keys and shapes you need depends on the policy/VLA and the dataset version, but the layout and API above are always the same. For our walkthrough the target schema is:

 raw (.npz)                          LeRobotDataset key            stored as
 ─────────────────────────          ──────────────────────        ─────────
 rgb     (T,64,64,3) uint8     ──▶   observation.images.front  ──▶ mp4   (video)
 state   (T,6)       float32   ──▶   observation.state         ──▶ parquet
 action  (T,6)       float32   ──▶   action                    ──▶ parquet
 task    "pick up the cube"    ──▶   "task" (per frame)        ──▶ tasks.parquet

Understand the Conversion Flow

Converting is always the same five steps:

  1. features      define the schema: per key -> dtype, shape, names
        |
        v
  2. create        LeRobotDataset.create(repo_id, fps, features, root, robot_type, use_videos=True)
        |                   
        v
  3. add_frame     for each timestep, add one row: add_frame({observation.*, action, "task": ...})
        |            ^ the "task" key is REQUIRED
        v            
  4. save_episode  call once at the end of each episode
        |
        v
  5. finalize      closes the parquet writers + writes metadata
        |            <- skip this and the dataset won't load
        v
  (optional) push_to_hub()   share on the Hub (needs network -> login node)

In code, the core of 03_dataset_conversion/convert_sample.py looks like this — read it as the template you will adapt in the exercise:

import numpy as np, glob
from lerobot.datasets.lerobot_dataset import LeRobotDataset

# 1. features — the schema. "video" -> encoded to mp4; numeric -> parquet.
features = {
    "observation.images.front": {"dtype": "video", "shape": (64, 64, 3), "names": ["height", "width", "channel"]},
    "observation.state": {"dtype": "float32", "shape": (6,), "names": ["state"]},
    "action": {"dtype": "float32", "shape": (6,), "names": ["action"]},
}

# 2. create — makes the meta/ data/ videos/ skeleton on disk
ds = LeRobotDataset.create(
    repo_id="local-user/handson-convert-sample",
    fps=10,
    features=features,
    root=".smoke/converted",
    robot_type="generic",
    use_videos=True,  # camera -> mp4  (False would store frames as images)
)

# 3 + 4. stream frames in, then seal each episode
for path in sorted(glob.glob(".smoke/raw/episode_*.npz")):
    z = np.load(path)
    task = "pick up the cube"
    for t in range(len(z["action"])):
        ds.add_frame({
            "observation.images.front": z["rgb"][t],  # (64,64,3) uint8
            "observation.state": z["state"][t],  # (6,) float32
            "action": z["action"][t],  # (6,) float32
            "task": task,  # <- required every frame
        })
    ds.save_episode()  # writes this episode's parquet rows + encodes its video

# 5. finalize — closes parquet writers and writes meta/info.json, stats.json, tasks.parquet
ds.finalize()

What does the work under the hood: LeRobot writes the tabular rows to Parquet (columnar, compressed), encodes each camera stream to MP4 with ffmpeg, and computes the normalization stats while it streams. You never touch pyarrow or ffmpeg directly — add_frame / save_episode / finalize do it.

Two things silently break a conversion: forgetting the "task" key in add_frame, and forgetting finalize() (the parquet stays open and the dataset won't load). convert_sample.py does both correctly.

4. Policy Training

Now we train a policy on a dataset. We use Action Chunking with Transformers (ACT) on a shared LIBERO dataset, with the same fixed step budget for everyone.

The dataset — LIBERO

LIBERO is a standard benchmark of language-conditioned tabletop manipulation tasks (pick/place, open/close…), simulated in robosuite/MuJoCo on a Franka Panda arm, grouped into task suites:

  • libero_object: same layout, different objects (object generalization)
  • libero_spatial: same objects, different placements (spatial reasoning)
  • libero_goal: same scene, different goals (goal generalization)
  • libero_10: 10 long-horizon tasks
  • libero_90: 90 short tasks

It is published on the Hub in LeRobot format ( $DATA_REPO , pre-downloaded). Each frame is one row:

observation.images.image: (H,W,3), uint8, agent-view (3rd-person) camera
observation.images.image2: (H,W,3), uint8, wrist (eye-in-hand) camera
observation.state: (8,), float, proprioceptive robot state
action: (7,), float, end-effector command (Δpose 6 + gripper 1)
task: str, "pick up the alphabet soup and place it in the basket"

These are the same keys ACT trains on and that the LIBERO simulator expects, so training and evaluation stay consistent. ACT ignores task (it is not a language model) and learns one suite at a time.

How training works

One command, lerobot-train, runs the whole loop.

  1. loads the LeRobotDataset and normalizes the observations and actions,
  1. builds the ACT policy from your --policy.* flags,
  1. runs a supervised loop — sample a batch, predict an action chunk, compare it to the demonstrated actions, back-propagate, step the optimizer,
  1. every so often, writes a checkpoint and logs the loss.

5. Policy Evaluation

Training gives you a checkpoint, but a low training loss does not tell you whether the policy actually works. You have to run it and see.
You could put it straight on a real robot, but that is slow, expensive, and risky — a bad policy can damage the arm or its surroundings. So you evaluate in a simulator first. The simulator runs the policy in closed loop and reports how often it finishes the task. This lets you check the policy and confirm that training went well, before any real hardware is involved.
So here we deploy the policy we trained in the LIBERO simulator, and measure how well it does.

        ┌──────────────────────────────────────────────┐
        ▼                                              │
   LIBERO sim renders obs (images + state)             │
        │                                              │ repeat each step,
        ▼                                              │ for N episodes
   your ACT checkpoint → action chunk                  │
        │                                              │
        ▼                                              │
   sim executes the action ────────────────────────────┘
        │
        ▼
   episode ends → success / failure   ──▶   success rate (%) over N episodes

The LIBERO simulator on HPC

The LIBERO env runs on robosuite/MuJoCo, so it needs GPU rendering: inside the container with --nv, and with MUJOCO_GL=egl (headless hardware rendering). The LIBERO package and its scene assets are baked into the image (env/build_image.sh) and work offline — nothing is downloaded at run time.

How to read the score

Low success is expected a 1000-step ACT scores low — that's the budget, not failure. The win is a working closed-loop eval, not a strong policy. Eval variance 10 episodes is noisy; a small gap may be noise, not a better model. Next step — real robots and the deploy system 👉

Today everything ran in a simulator. The next thing to understand is the other side: running a policy on a real robot, and the deploy system around it.

  • Real hardware. Sim (LIBERO) skips contact, latency, and perception noise. Closing the sim-to-real gap is its own discipline.
  • Deployment. A trained checkpoint is not a working robot. You need the loop that reads the cameras and state, runs the policy at control rate, and sends actions to the arm — at high enough Hz, with the async inference idea from SmolVLA so the robot does not stall waiting for the next chunk.

Understanding and building that loop — not just the training script — is what turns a checkpoint into something that acts in the world.

Where this is going — close the loop yourself 😎

The big picture, and what I want you to take away.

Robot learning runs more and more on VLAs (π0, SmolVLA, GR00T). To make progress with them, what matters is being able to test a model fast — and to build the system that lets you do that.

The full real-world loop is expensive:

 collect data ──▶ pretrain ──▶ finetune ──▶ deploy
       ▲                                      │
       └──────────────────────────────────────┘
       one turn = many robots, time, and compute

One turn costs a lot — many robots, people, time, and compute. For a single lab, running the whole loop end to end is about as much as you can afford.

But you don't need the whole loop to do real research. A lot of the open problems with VLAs can be studied in simulation, with a small setup:

  • the out-of-the-box (OOB) gap — pretrained generalists (π0, GR00T) often do poorly on a new task before adaptation.
  • how to close it — better data, light fine-tuning, RL in simulation or a digital twin.

You can study these yourself. Take an existing model, run it in a simulator, change one thing, measure. No robot fleet needed.

That's the direction I want to leave you with. You now have the full minimal workflow — convert, train, evaluate, all in simulation. The expensive parts will always be there, but the interesting questions are closer than they look. Pick one, start an interactive GH200, and go.
